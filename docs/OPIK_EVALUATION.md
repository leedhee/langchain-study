## 1. 평가 구성

### 1-1. 평가 목적

이번 평가는 의료 정보 안내 에이전트가 **질문에 적절히 답변하는지**, 그리고 **필요한 도구를 올바르게 선택하고 사용하는지**를 확인하기 위해 Opik을 활용해 진행.

### 1-2. 평가 지표

평가에는 `AnswerRelevance`, `ScorerWrapperMetric`, `Usefulness`를 활용.

- `AnswerRelevance`  
  사용자 질문과 최종 답변의 관련성을 확인하는 지표
- `ScorerWrapperMetric`  
  `AgentToolCorrectnessJudge`를 감싼 metric으로, 어떤 도구를 선택했고 해당 도구를 적절하게 사용했는지 확인하는 지표
- `Usefulness`  
  답변이 실제 사용자 입장에서 도움이 되는지, 필요한 설명이 충분한지 확인하는 지표

### 1-3. 데이터셋 구성

데이터셋은 **증상, 약, 병원 검색 시나리오**를 기준으로 직접 구성.

단순 질의뿐 아니라 **복합 질의**도 함께 포함해, 실제 사용 상황에 가까운 형태로 설계.

- 단순 질의: 하나의 의도만 포함한 질문
- 복합 질의: 증상+약, 증상+병원, 병원+약처럼 둘 이상의 요청이 함께 포함된 질문

각 항목에는 사용자 입력과 `expected_output`을 함께 정의.
정답 문장을 그대로 맞추기보다, **핵심 내용을 담고 있는지**, 그리고 **안전한 답변 방향을 유지하고 있는지**를 중심으로 평가.

## 2. 이슈 및 개선 사항

### 2-1. 자유서술형 증상 질의 관련 이슈 

#### 이슈

자유서술형 증상 질문에서는 응답 품질이 흔들리는 문제가 있었음.

예시 : `속이 쓰리고 아픈데 위염 증상일 수도 있어?`
위와 같은 질문은 자연스럽지만, 검색 관점에서는 핵심 키워드가 분산되면서 검색 결과와 응답 품질이 흔들렸음

현재 증상 검색은 **BM25 기반 키워드 중심 검색 구조**로 동작하기 때문에, 핵심 증상이나 질환명이 분명한 질문에서는 비교적 안정적이지만 자유서술형 질문에서는 검색어가 분산되기 쉬운 특성이 있었음.

#### 개선사항

먼저 현재 검색 구조에 맞는 형태로 평가 데이터셋 질의를 조정.

- 기존 표현: `속이 쓰리고 아픈데 위염 증상일 수도 있어?`
- 수정 표현: `속쓰림과 복통이 있는데 위염 증상일 수 있어?`

의미는 유지하면서도 핵심 증상과 질환명이 보다 명확히 드러나도록 수정해 현재 검색 구조에 맞게 평가 일관성을 높임.

또한 검색 로직도 함께 보완.
질문 안에 질환 후보가 포함된 경우 해당 질환 키워드를 우선 검색하도록 변경했고, 특정 질환명을 하드코딩하는 대신 **정규식 기반으로 질환 후보를 추출**하는 방식 적용.

즉, 현재 단계에서는 BM25 기반 검색 구조에 맞게 질의를 일부 정리하고, 질환 키워드 중심으로 검색이 이루어지도록 보완.

실제 적용 코드 예시는 아래와 같음.

```python
# 질환명과 같은 후보군 추출을 위한 정규식 
DISEASE_HINT_PATTERN = re.compile(
    r"([가-힣A-Za-z][가-힣A-Za-z0-9]*(?:"
    r"염|증|질환|증후군|"
    r"암|종양|궤양|결석|출혈|폐색|마비|손상|골절|감염|비대|"
    r"역류질환|기능저하증|항진증|저혈당|고혈압|저혈압))"
)

def _build_symptom_search_queries(symptom_name: str) -> list[str]:
    # 원문 질의를 정리한 뒤 질환 후보와 증상 키워드 분리 
    normalized_query = _normalize_symptom_query(symptom_name)
    disease_hint = _extract_disease_hint(normalized_query)
    symptom_terms = _split_symptom_terms(normalized_query, disease_hint)

    search_queries: list[str] = []
    # 질환 후보가 있으면 해당 키워드를 우선 검색어로 사용
    if disease_hint:
        search_queries.append(disease_hint)

    # 질환 후보와 증상 키워드가 모두 있으면 조합 검색어 추가 
    if disease_hint and symptom_terms:
        search_queries.append(" ".join([disease_hint, *symptom_terms]))
    elif symptom_terms:
        # 질환 후보가 없으면 증상 키워드만으로 검색
        search_queries.append(" ".join(symptom_terms))

    # 위 단계에서 검색어가 만들어지지 않으면 정규화된 원문을 fallback으로 사용
    if not search_queries and normalized_query:
        search_queries.append(normalized_query)

    return search_queries
```

### 2-2. 반복 도구 호출 제어

#### 이슈

일부 질문에서는 비슷한 의미의 재검색이 반복되면서 **도구 호출 횟수가 과도하게 늘어나는 문제**가 있었음.

이로 인해 응답이 길어지거나 불안정해질 수 있었고, 불필요한 호출 비용도 발생할 수 있었음.

#### 개선사항

이를 줄이기 위해 프롬프트에 **반복 재검색 제한 규칙**을 추가.
현재는 한 번의 사용자 요청에서 도구 호출이 과도하게 이어지지 않도록 제어하는 방식으로 안정성을 보완한 상태. 

다만 이 방식은 구조적인 해결이라기보다, 현재 에이전트 흐름에서 반복 호출을 제어하기 위한 운영상 보완 장치에 가까움.
향후에는 LangGraph 기반 분기 처리를 적용해, 프롬프트 제어보다 흐름 제어 중심으로 개선할 계획.

실제 적용 코드 예시는 아래와 같음.

```text
- 이미 확보한 정보로 답변 가능하면 추가 호출 없이 최종 답변을 생성한다.
- 같은 의미의 재검색이나 불필요한 반복 호출은 하지 않는다.
- 한 번의 사용자 요청에서 도구 호출은 최대 3회 이내로 마무리한다.
```

## 3. 평가 결과 요약

### 3-1. 평균 점수

- `AnswerRelevance`: `0.918`
- `tool_correctness`: `0.960`
- `Usefulness`: `0.647`

### 3-2. 결과 해석

전체적으로 보면 질문과 답변의 관련성은 높은 수준이었고, 도구 선택과 사용도 비교적 안정적으로 이루어졌음.

반면 `Usefulness`는 다른 지표에 비해 상대적으로 낮게 나타났음.
답변 자체는 크게 틀리지 않았지만, 사용자가 보기에 더 바로 이해하기 쉽고 실용적으로 느껴지도록 보완할 부분이 남아 있음

즉, 현재 **질문 적합성과 도구 사용은 비교적 안정적이지만, 최종 답변의 실용성과 간결성은 추가 개선이 필요한 상태**

## 4. 한계 및 향후 계획

### 4-1. 현재 한계

현재 검색 구조가 **BM25 기반 키워드 중심 방식**이기 때문에, 
자유서술형 질문이나 표현이 다양한 입력에 대해서는 대응력이 충분하지 않을 수 있음.

또한 검색 결과가 문서 원문 중심으로 길게 반환되는 경우가 많아, 이를 그대로 프롬프트에 포함하면 input token이 증가하고 응답 생성 효율이 떨어질 수 있음.
따라서 retrieval 이후 **결과 압축과 요약 과정**이 추가로 필요.

### 4-2. 향후 계획

3주차에는 **하이브리드 검색**을 도입해 자유서술형 질문 대응력을 높일 계획.
또한 하이브리드 검색 적용 이후에는 현재 사용 중인 질의 정규화 로직을 최소화하거나 보조 전처리 수준으로 축소할 예정.

- retrieval 결과 압축 및 요약 보완
- 평가 데이터셋 확장
- `Usefulness` 관점의 답변 실용성 개선
- LangGraph 기반 분기 처리 적용을 통한 반복 도구 호출 구조 개선
